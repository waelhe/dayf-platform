## القرارات المؤسسة (Core Decisions)

---

### ADR-001: نظام Escrow كطبقة ثقة مالية

**التاريخ:** 2025
**الحالة:** ✅ مُطبق

**القرار:** استخدام Escrow مدمج بدلاً من الدفع المباشر للمضيف

**السبب:** السياحة السورية تحتاج ضماناً مالياً قوياً بسبب غياب الثقة التاريخي بين السياح والمزودين.

**المقايضة:** تعقيد أكبر في flows المالية، لكن ثقة أعلى = تحويل أعلى.

**الأثر:** كل نظام حجز يمر عبر Escrow — لا يمكن تجاوزه.

---

### ADR-002: SQLite في مرحلة MVP

**التاريخ:** 2025
**الحالة:** 🟠 قيد التغيير — الهجرة إلى Supabase

**القرار:** SQLite + Prisma للـ MVP

**السبب:** سرعة التطوير، لا حاجة لإعداد database server، deployment أبسط.

**المقايضة:** أداء أقل للـ concurrent writes، لا horizontal scaling.

**شرط التغيير:** عند تجاوز 500 مستخدم نشط يومياً أو أي مشكلة locking واضحة.

**التالي عند التغيير:** PostgreSQL + Supabase (البنية الحالية تدعم التغيير بدون تعديل application layer بفضل Prisma).

---

### ADR-003: الأنظمة الخمسة الذكية كوحدات مستقلة

**التاريخ:** 2025
**الحالة:** ✅ مُقرر — قيد التنفيذ

**القرار:** كل نظام من الأنظمة الخمسة = module مستقل يتواصل عبر Event Bus فقط.

**السبب:** المنصة ستنمو — الاستقلالية تتيح تطوير كل نظام بدون كسر الآخرين.

**المقايضة:** overhead في تصميم الأحداث، debugging أصعب.

**القيد:** لا استدعاء مباشر بين modules — هذا القرار ثابت (المادة V من الدستور).

---

### ADR-004: نظام المراجعة أولاً (Phase 15)

**التاريخ:** 2025
**الحالة:** 🟠 مُقرر — لم يبدأ التنفيذ

**القرار:** نظام المراجعة الذكي هو الفيز التالي.

**السبب من التشخيص:**
- هو المحور الذي تعتمد عليه الأنظمة الأربعة الأخرى
- البنية جاهزة 85% — أقل جهداً
- Escrow + Booking موجودان = Verified reviews متاحة فوراً
- بدونه: نظام التوصية والـ Gamification لا يعملان

**ترتيب التنفيذ داخل النظام:**
1. نماذج Prisma (ReviewCriteria, ReviewHelpful, ReviewerBadge)
2. Review creation flow بعد booking.completed
3. عرض المراجعات على Listings
4. نظام التصويت على المفيد/غير المفيد
5. خوارزمية الترتيب الكلاسيكية
6. إضافة طبقة AI (Sentiment + Summary)

---

### ADR-005: AI Provider عبر z-ai-web-dev-sdk

**التاريخ:** 2025
**الحالة:** ✅ مُطبق (SDK موجود)

**القرار:** كل استدعاءات AI تمر عبر `z-ai-web-dev-sdk` كـ abstraction layer.

**السبب:** تغيير provider (OpenRouter → Anthropic → Google) بدون لمس application code.

**القيد:** لا استدعاء مباشر لأي AI API من داخل application layer.

**Fallback إلزامي:** كل service تستخدم AI يجب أن تعمل بنتائج أقل جودة بدونه.

---

### ADR-006: Socket.io للـ Real-time

**التاريخ:** 2025
**الحالة:** 🟡 مُقرر — مؤجل إلى Phase Chat

**القرار:** Socket.io للـ real-time (Chat + Notifications + Check-in tracking).

**السبب:** Next.js يدعمه، Team familiar به، يتكامل مع الـ infrastructure الحالي.

**المقايضة:** لا يدعم horizontal scaling بدون Redis adapter.

**شرط التفعيل:** عند بناء Chat feature في نظام التجربة المتكاملة.

**ملاحظة:** Socket.io server يعمل في `/app/api/socket/route.ts` — لا custom server.

### PDR-001: خوارزمية التوصية — أي نهج؟

**التاريخ:** 2025
**الحالة:** ⏳ قيد النقاش

**السؤال:** Collaborative Filtering vs Content-Based vs Hybrid LLM?

**الخيارات:**

| الخيار | الإيجابيات | السلبيات | متطلبات البيانات |
|--------|-----------|---------|-----------------|
| Content-Based | يعمل بدون بيانات سلوكية | محدود الاكتشاف | بيانات Listing فقط |
| Collaborative Filtering | اكتشاف ممتاز | يحتاج cold-start solution | سجل حجوزات كافٍ |
| LLM-based | مرن + يفهم النص | تكلفة + latency | أي بيانات |

**توصية مؤقتة:** ابدأ بـ Content-Based (أسرع بناءً، لا cold-start problem)، انتقل لـ Hybrid بعد 1000 حجز.

**القرار المطلوب من:** Wael — قبل بداية Phase تخطيط السفر.

### PDR-002: Gamification — نظام النقاط

**التاريخ:** 2025
**الحالة:** ⏳ قيد النقاش

**السؤال:** كيف تُحسب نقاط الولاء؟ وما العلاقة مع Trust Score؟

**الاقتراح الأولي:**
```
نقاط الحجز  = قيمة الحجز بالدولار × 0.1
نقاط المراجعة = 50 (verified) + bonus جودة
نقاط التصويت = 5 لكل helpful vote
نقاط المجتمع = 3 لكل reply مفيد
```

**القرار المطلوب من:** Wael — قبل بداية Phase Gamification.

### PDR-003: الهجرة إلى Supabase

**التاريخ:** 2025-03
**الحالة:** 🟠 قيد التنفيذ

**السؤال:** كيف نهاجر من SQLite إلى Supabase بدون تعطيل التطبيق؟

**الخطة:**
1. إنشاء طبقة تجريد قاعدة البيانات (Repository Pattern)
2. بناء SupabaseRepository
3. هجرة 26 خدمة من Prisma
4. إضافة Auth protection لـ 53 route
5. إضافة Zod validation لـ 51 route

**القيد:** تطبيق يعمل أثناء الهجرة — لا downtime.
